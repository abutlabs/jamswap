# JAMswap service observability & load plan

**Goal:** a Grafana page that answers, at a glance, "is the flagship JAM service
healthy, how fast do orders settle, and where in the pipeline are they stuck?" —
plus a load-injection harness that turns those dashboards into a performance
instrument. Scoped 2026-07-08, motivated by the faucet incident below.

---

## The motivating incident (and why it wasn't a bug)

On a freshly relaunched `make mixed` chain, the UI showed **"Registration
pending — click your account again in a moment"** three times and the faucet
was unusable. Diagnosis:

- **No accumulate error.** The register work-items were queued, guaranteed,
  and settled — Alice's pubkey got handle 2 a few minutes later. The chain was
  simply in the launch warm-up (AURA→ticket transition + DEX bootstrap items
  contending for the C=2 cores), where settlement latency spikes; the doc's
  steady-state caveat in `MIXED_CHAIN_DEX_SETTLEMENT.md` describes exactly this.
- **The real gap is visibility.** The UI polls the handle once and gives up
  with a toast; the operator has no way to see "your register is queued on 3
  nodes, guaranteed at slot S, waiting for an assurance". Nothing measures
  submit→settle latency, so "slow" and "broken" are indistinguishable.

That gap is what this plan closes.

## Phase 0 — registration/faucet UX (small, do first)

1. **UI auto-poll**: after `/api/register`, poll `/api/handle` with backoff
   (2s → 5s, up to ~3 min), progress toast ("registering… slot N, waiting for
   settlement"), and disable duplicate submits per pubkey. Same pattern for
   deposit/withdraw: optimistic "pending" row until the balance moves.
2. **`/api/pending`**: the dex server keeps an in-memory ledger of relayed
   submissions `{op, key, submitted_at, settled_at|null}` (it already knows the
   settle condition per op — it polls the same reader). The UI and the canary
   read it; it is also the source for the settle-latency histogram in Phase 1.
3. **Builder `/submit` response**: include per-target accept/fail counts
   (it already fans out; surface what happened).

Deliverable: no more dead-end "click again" toast; registration visibly
progresses and self-completes.

## Phase 1 — instrument the whole pipeline

One new metric surface per component, all scraped by the existing Prometheus:

1. **dex `server.py` `/metrics`** (stdlib text format, no new deps):
   - `jamswap_api_requests_total{route,code}`, `jamswap_api_errors_total`
   - `jamswap_submits_total{op}` (register/deposit/withdraw/cancel/order/round)
   - `jamswap_settled_total{op}`, `jamswap_settle_timeouts_total{op}`
   - `jamswap_settle_latency_seconds{op}` histogram — **the headline SLO
     metric**, measured submit→state-visible via the Phase-0 pending ledger
   - service-state gauges: registered accounts, markets, book depth per market,
     treasury JAMKB reserve vs target, storage footprint octets
2. **builder `/metrics`**: `builder_submits_total{target}`,
   `builder_submit_failures_total{target}`, CE-133 ack latency histogram.
3. **lasair** (one small client change): gauge `lasair_ce133_queue_depth`,
   counter `lasair_ce133_assured_cores_total` (assure-any-pending activity),
   and `lasair_ce133_accumulated_total` (the "work-item accumulated" event —
   today it's only a log line). Everything else needed already exists.
4. **prometheus.yml**: add `dex:8080` and `builder:19980` scrape jobs.
5. **Canary**: a `canary` container running a slimmed `verify.py` (register +
   deposit only, fresh account) every 5 min, exporting
   `jamswap_canary_{pass,fail}_total` + duration — the end-to-end heartbeat
   that catches what unit metrics miss.

## Phase 2 — the JAMswap dashboard (`jam-service.json`)

Generated via the existing `gen_dashboards.py` pattern; becomes the fourth
provisioned dashboard and links to/from `jam-mixed`. Rows:

1. **Order funnel** (single-stat + rate row): submitted → queued (Σ lm nodes)
   → guaranteed → accumulated; funnel conversion %, settle success SLO
   (settled / submitted over 15 min).
2. **Settlement latency**: p50/p95/p99 per op; latency in *slots* (÷6 s);
   canary duration trend. Annotation stream from chain warm-up (first 2 epochs
   shaded — expected-slow region).
3. **Pipeline mechanics** (the tier-1 fix, visible): guarantees formed vs
   `lost fork choice` re-queues (orphan ratio), duplicate drops (fan-out
   working), `no free core` stalls (core contention), queue depth per node,
   time-in-awaiting.
4. **Chain context** (imported panels/links): authoring split lasair:pj, fork
   rate, ticket vs fallback seals, height/slot — so a settlement stall is
   immediately correlatable with a consensus event.
5. **Service state**: accounts, balances by asset, treasury reserve vs rent
   target, footprint, markets/book depth.
6. **API & bridges**: dex request/error rates, reader CE-129 latency/errors,
   builder per-target failures (a down lm node shows here first).
7. **Alert rules** (Grafana-native): canary 2 consecutive fails; settle p95 >
   10 slots for 10 min; orphan ratio > 80% sustained; builder target down.

## Phase 3 — load injection (`loadgen`)

A compose service (Python asyncio, no framework) that drives the public DEX
API with configurable profiles — measurements taken from the *outside* so the
dashboards are validated against an independent observer:

- **Profiles** (env-selected): `steady` (r ops/s mixed), `burst` (idle → spike),
  `soak` (hours at moderate rate), `faucet-storm` (mass registers — the
  incident scenario), `trading-day` (orders/cancels/matches on all markets).
- N virtual accounts pre-registered at ramp-up; every op stamped and tracked to
  settlement; exports its own `/metrics` (offered load, in-flight, e2e settle
  latency, timeouts) → same dashboard, "Load" row.
- **Capacity note to validate**: the hard ceiling is **C=2 cores × 1 report per
  core per block ≈ 2 work-items/block ≈ 0.33 items/s**, halved-ish by fork
  churn on the equal split; the builder currently wraps ONE work-item per
  package. **Batching multiple queued items into one package** is the obvious
  throughput lever the load tests will motivate — measure first, then decide.
- **Success criteria** (initial SLOs to confirm/adjust empirically):
  - settle p95 ≤ 8 slots (48 s) at 0.2 ops/s steady, post warm-up
  - zero *lost* ops at any offered rate (latency may grow, ops must not vanish)
  - orphan recovery 100% (every `lost fork choice` re-queue eventually settles)
  - canary green across a 24 h soak

## Phase 4 — Kubernetes (optional, after 1–3 prove the metrics)

Honest scoping: compose is sufficient for the 6-validator devnet and a single
loadgen. k8s earns its complexity when we want (a) horizontally scaled load
generators, (b) a long-lived remote soak environment, (c) **chaos experiments**
— kill a pj validator mid-load, inject latency between authors (netem), and
watch the funnel/orphan panels respond. Plan when we get there: kind/k3d
manifests (or a small helm chart) mirroring the compose topology, shared
Grafana provisioning, chaos-mesh for fault injection. Not before.
